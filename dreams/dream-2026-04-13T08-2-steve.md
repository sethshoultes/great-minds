## Idea 1: Witness

### One-liner
A Claude Code skill that watches your coding session and produces a 60-second narrated video showing what you built — using Remotion and TTS to turn invisible work into shareable proof.

### The User
Developers who ship constantly but have nothing to show for it. Their GitHub activity graph is green but their portfolio is empty. They build remarkable things and then... nothing. The work vanishes into a commit hash. Their mom doesn't know what they do. Their clients can't see the effort. They're invisible craftspeople in a world that only sees the final product.

Witness is for every developer who has ever wished someone was watching when they solved that impossible bug at 2am.

### Distribution
- Claude Code skill — ships through the great-minds-plugin ecosystem we already have
- Zero friction: `/witness` after any session generates the video
- "I built this" is the most shareable content format on developer Twitter/X
- Launch post: "We let an AI watch us code for a week — here's the 60-second movie of everything we shipped"
- The output is designed to be embedded in portfolios, proposals, and progress reports

### Why It's Insanely Great
Every other dev tool helps you build. Witness helps you *show*.

The video opens with a code diff animation — files changing, functions appearing, tests passing. A calm AI voice narrates: "In this session, the developer implemented a caching layer that reduced API calls by 73%. They started with a failing test, iterated through three approaches, and landed on a solution that will save their users approximately 2 seconds per page load."

This isn't a screen recording. It's a *story*. The AI watches the git diff, understands the intent, and produces something a human would be proud to share. Parents finally understand what their kid does. Clients see the craft behind the invoice. The work becomes real.

The magic is the compression: hours of effort become a single compelling minute. That's what makes it shareable. That's what makes it matter.

### Build Plan
**Steve:** Defines the video narrative arc — what makes a 60-second dev video compelling. Writes the prompt that turns diffs into stories. Designs the visual language: what the code animations look like, what the voiceover sounds like, what the pacing feels like.

**Elon:** Engineers the Remotion integration — git diff parsing, code animation components, TTS API calls, video rendering pipeline. Defines the data model for sessions and the slash command interface.

**Sub-agents:**
- Remotion Engineer: Video composition, code-diff animation components, timeline assembly
- Prompt Engineer: Diff-to-narrative prompt chain, voice calibration for different session types
- Designer: Visual style guide for code animations, typography, pacing templates

**Session flow:** Define narrative format -> build diff parser -> design Remotion components -> TTS integration -> slash command scaffold -> test on real session -> ship

### Dogfood Factor
Maximum. Every time the agency ships, we run `/witness`. The Great Minds blog becomes a video portfolio of everything we build. Our clients see the work happen. This is infrastructure for showing our craft.

## Idea 2: Tempo

### One-liner
A WordPress plugin that turns your blog posts into narrated video content using AI voiceover and simple motion graphics — one click, zero editing skills required.

### The User
WordPress creators who know they should be on YouTube but can't face the camera or learn video editing. They write great content and watch it get buried by the algorithm while video dominates every platform. They see creators with millions of views and think "I could never do that" — not because they lack ideas, but because they lack the tools.

Tempo is for the blogger who writes better than they speak and wants their words to travel.

### Distribution
- WordPress.org plugin directory (free tier for text-only, premium for custom voices)
- Natural upgrade path from Dash, Pinned — our existing WP ecosystem
- Content creators are always looking for repurposing tools
- YouTube shorts, TikTok, Instagram Reels — one post becomes four formats
- Launch video: a blog post about Tempo, turned into a Tempo video

### Why It's Insanely Great
Video isn't replacing text — video is *eating* text. But most bloggers can't make the leap because video feels like a different skill. It isn't. The skill is having something to say. Tempo handles the rest.

You write your post. You click "Create Video." Tempo analyzes your content, extracts the key narrative beats, generates a script optimized for voiceover, renders it with your choice of AI voice, and produces a 60-90 second video with elegant text animations and a subtle soundtrack.

The output isn't a slideshow of bullet points. It's a *video*. It has pacing. It has visual hierarchy. It has a beginning, middle, and end. It respects the craft of video while honoring the craft of writing.

The person who writes a great blog post shouldn't need to become a different person to share that post with the YouTube generation.

### Build Plan
**Steve:** Defines the video aesthetic — what makes a text-to-video feel like a real video, not a slideshow. Writes the prompts that extract narrative from posts. Chooses the visual language: typography, motion, color.

**Elon:** Engineers the WordPress integration — post content extraction, Claude API for script generation, Remotion template rendering, video storage and delivery. Defines the admin UI and rendering queue.

**Sub-agents:**
- WordPress Engineer: Plugin scaffold, admin interface, post meta integration
- Remotion Engineer: Video template library, text animation components, audio sync
- Prompt Engineer: Post-to-script conversion, voice tone calibration

**Session flow:** Define video format -> WordPress plugin scaffold -> script extraction pipeline -> Remotion templates -> TTS integration -> admin UI -> test on real posts -> ship

### Dogfood Factor
High. Every Great Minds blog post becomes a video. We test the format on our own content, iterate based on real results, and use it to grow our own audience. The product improves every time we publish.

## Idea 3: Shrine

### One-liner
A command-line tool that turns any GitHub repo into a beautiful, shareable one-page website — your README as a product page, generated in seconds, hosted instantly.

### The User
Open-source maintainers and indie developers who have built something remarkable but haven't built a marketing page. Their README is comprehensive but ugly. Their project deserves attention but looks amateur. They know they should have a landing page but they don't want to become a web designer.

Shrine is for the developer who believes the code should speak for itself — and then watches it speak to an empty room.

### Distribution
- CLI tool: `npx shrine` in any repo
- GitHub Action: auto-update the site on every push
- Natural fit for the Claude Code ecosystem — mention it in skills, recommend it after project completion
- Open source the renderer, freemium hosting model
- Launch: "We shrine'd 100 popular repos without permission — here's how they look now"

### Why It's Insanely Great
A beautiful landing page is table stakes. But developers hate making them. So they don't. And then they wonder why nobody uses their project.

Shrine reads your README, understands your project, and generates a single-page website with: a hero section featuring your one-liner, a problem/solution narrative extracted from your docs, feature highlights with clean iconography, installation instructions that actually look inviting, a contributor section, and a call to action.

The design is opinionated and beautiful. No configuration. No customization paralysis. You run the command, you get a site. If you don't like it, edit your README and run it again.

The insight is that READMEs already contain everything a landing page needs — they just don't *look* like landing pages. Shrine bridges that gap in 30 seconds.

### Build Plan
**Steve:** Defines the design system — the one template that makes every project look great. Writes the prompts that extract marketing copy from technical README content. Decides what's shown and what's hidden.

**Elon:** Engineers the CLI — README parsing, Claude API for copy extraction, static site generation, Vercel/Cloudflare Pages deployment. Defines the GitHub Action for continuous deployment.

**Sub-agents:**
- CLI Engineer: npx scaffold, README parsing, deployment integrations
- Prompt Engineer: README-to-marketing extraction, section identification
- Designer: The single beautiful template that works for everything

**Session flow:** Define template -> build README parser -> Claude extraction prompts -> static site generator -> deployment integration -> GitHub Action -> test on real repos -> ship

### Dogfood Factor
Total. Every one of our repos gets a shrine. The great-minds-plugin, LocalGenius, Dash, Pinned — all of them become beautiful landing pages. We use it to launch every future project. This is how we present our work.

## My Verdict

These three products share a common insight: **developers create invisible value and have no way to make it visible**.

**Witness** makes the work visible through video. **Tempo** makes the writing visible on platforms that ignore text. **Shrine** makes the code visible through marketing design.

If I had to pick one to build today, it's **Witness**. It's the purest expression of what we can do — multi-agent orchestration meets video production meets developer tooling. It ships through our existing plugin infrastructure. It produces output so shareable that it markets itself. And it solves a problem that every developer feels but nobody talks about: the loneliness of shipping to an empty room.

But I'd build all three. They're a family. The question is sequence.

— Steve
